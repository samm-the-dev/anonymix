# 02.2-Auth / Sign Up

## Page Metadata

| Property | Value |
|----------|-------|
| **Scenario** | 02: The Casual's First Session |
| **Page Number** | 02.2 |
| **Platform** | Mobile |

## Overview

**Page Purpose:** Create an account by signing up with a music platform. One OAuth flow handles both auth and platform linking — no separate platform connection step.

**Entry Context:** Tapped Join on Invite Landing (02.1)

**Exit Action:** Account created + music platform linked → auto-joined to session → Profile / Account (02.3, formerly 02.4)

## Design Decisions

- **Auth IS platform link:** "Sign up with Spotify" or "Sign up with YouTube Music" creates the account AND connects the music platform in one OAuth flow. No separate Platform Link step needed.
- **Platform Link step (02.3) eliminated:** Merged into auth. Adding/switching platforms happens on Profile page.
- **Name + avatar pulled from platform:** OAuth provides display name and profile image. User can customize later on Profile page.
- **One tap + OAuth redirect:** The Casual taps their platform, does the OAuth dance, comes back fully set up. Under 30 seconds.
- **Platform choice matters:** The user picks which music service they use — this determines their search catalog and playlist delivery in the game loop.

## Content Needs

- "Sign up with Spotify" button
- "Sign up with YouTube Music" button
- Minimal context: maybe session name reminder so they know they're still joining
- No email/password option — music platform OAuth only (ensures platform is always linked)

## Driving Forces Addressed

- **Effortless participation (14):** One tap to sign up. OAuth does all the work.
- **Fear of too many steps (14):** One page, two buttons, pick your platform. No forms, no passwords, no extra screens.

## On-Page Interactions

- Tap a platform button → OAuth redirect → return with account + platform linked
- No other interactions

## Complexity Notes

- Low UI complexity — two buttons
- OAuth integration with Spotify and YouTube Music APIs (existing infra from OHM)
- Need to handle OAuth failure gracefully (redirect back with error state)
- Opens in browser (from invite link flow) — OAuth redirect must work in mobile browser context
